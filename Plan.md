## 2. Перечень используемых инструментов с обоснованием выбора

| Инструмент | Версия | Назначение | Обоснование |
|---|---|---|---|

## 3. Перечень и описание возможных рисков при автоматизации

| № | Риск | Вероятность | Влияние | Меры снижения |
|---|------|-------------|---------|---------------|
| R-01 | Нестабильность UI-локаторов (изменение вёрстки приложения) | Средняя | Высокое | Использовать семантические селекторы (`data-testid`, роли), избегать XPath по структуре |
| R-02 | Асинхронность уведомлений банка — тест не дождётся ответа | Средняя | Высокое | Настроить `Configuration.timeout` на достаточное значение (10 с+), явные ожидания в методах страниц |
| R-03 | Различное поведение MySQL и PostgreSQL (типы данных, регистр статусов) | Средняя | Среднее | Параметризовать запуск тестов через системные свойства (`-Ddb.url`), прогонять оба профиля в CI |
| R-04 | Эмулятор gate-simulator не стартует или падает в ходе прогона | Низкая | Высокое | Добавить healthcheck в `docker-compose.yml`; в `@BeforeAll` проверять доступность порта 9999 |
| R-05 | Тесты оставляют «мусор» в БД, влияя на последующие проверки | Высокая | Среднее | Очистка таблиц в `@BeforeEach` через `SQLHelper.cleanDatabase()` |
| R-06 | Тесты падают из-за гонки состояний (быстрый UI + медленная БД) | Средняя | Среднее | Добавить небольшую задержку или polling при чтении статуса из БД |
| R-07 | Chrome/ChromeDriver версия несовместима с окружением | Низкая | Среднее | Зафиксировать версию через WebDriverManager или использовать headless-режим в CI |
| R-08 | Приложение `aqa-shop.jar` недоступно при запуске тестов | Низкая | Критическое | Описать в README порядок запуска; добавить проверку `http://localhost:8080` перед тестовой сессией |

| День 1–2 | Настройка окружения и инфраструктуры | Работающий `docker-compose.yml`, запускающийся `aqa-shop.jar` |

## 6. Ожидаемые результаты прогона

- **Всего тест-кейсов:** 35 (18 Payment + 17 Credit)
- **Ожидаемых PASS:** ~30 (позитивные + явные валидации)
- **Потенциально FAIL (предполагаемые баги приложения):** ~5  
  *(Например: некорректное сообщение об ошибке, данные карты попадают в БД, отличающееся поведение на PostgreSQL)*
- Все найденные дефекты оформляются в **GitHub Issues** с шагами воспроизведения, ожидаемым и фактическим результатом.
