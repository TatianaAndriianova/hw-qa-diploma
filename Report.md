# Отчёт о проведённом тестировании

## Краткое описание

В рамках дипломного проекта было проведено автоматизированное тестирование веб-сервиса покупки тура. Сервис предоставляет два способа оплаты: оплата дебетовой картой (Payment Gate) и выдача кредита по данным банковской карты (Credit Gate). Тестирование охватывало UI-взаимодействие, взаимодействие с банковскими эмуляторами, а также проверку корректности записей в базе данных.

Тесты были запущены в двух окружениях:
- **PostgreSQL** — в качестве СУБД;
- **MySQL** — в качестве СУБД.

## Процент успешных и неуспешных тест-кейсов

| Окружение  | Успешных | Упавших |
|------------|:--------:|:-------:|
| PostgreSQL | 62,9%    | 37,1%   |
| MySQL      | 62,9%    | 37,1%   |

Результаты идентичны в обоих окружениях, что свидетельствует об одинаковом поведении приложения при работе с MySQL и PostgreSQL.

---

## Общие рекомендации

1. **Исправить баги, приводящие к падению тестов.** 37,1% тест-кейсов завершились неудачей в обоих окружениях. Все найденные дефекты зафиксированы в виде Issues в репозитории проекта со скриншотами и описанием.

2. **Проверить корректность статусов платежей.** Часть упавших тестов связана с тем, что приложение возвращает некорректный статус операции (например, APPROVED вместо DECLINED или наоборот) — это критичный дефект для платёжного сервиса.

3. **Проверить запись данных в БД.** Ряд тестов выявил расхождения между фактическим статусом платежа и значением, сохранённым в базе данных. Необходимо убедиться в атомарности и корректности транзакций.

4. **Унифицировать поведение для MySQL и PostgreSQL.** Несмотря на то что результаты тестов совпали, рекомендуется регулярно проводить параллельное тестирование на обеих СУБД, чтобы своевременно выявлять расхождения.

5. **Расширить тестовое покрытие.** После исправления текущих дефектов рекомендуется добавить тест-кейсы на граничные значения полей формы, а также на поведение сервиса при недоступности банковского эмулятора.
